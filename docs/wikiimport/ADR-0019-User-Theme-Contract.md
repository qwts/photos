# ADR-0019: User Theme Contract — Token Surface, Validation, and Safety

## Status

Accepted 2026-07-17 on issue [#394](https://github.com/qwts/photos/issues/394) (proposed and owner-accepted the same day; any section may still be amended by owner veto before its implementing code lands). This ADR extends [ADR-0001](ADR-0001-Automation-Check-Governance) (gates) and coordinates with the accessibility tooling from #381; it rewrites nothing.

Section map for the epic's children: §2, §5, §7 govern [#395](https://github.com/qwts/photos/issues/395) (application plumbing + light theme); §1, §3, §4, §5 govern [#396](https://github.com/qwts/photos/issues/396) (theme manager); §6 governs [#397](https://github.com/qwts/photos/issues/397) (template export + LLM docs).

## Context

The renderer is dark-only by construction: five token files define flat custom properties on `:root` in oklch (`src/renderer/src/styles/tokens/`), with semantic aliases (`--surface-*`, `--text-*`, `--selection`, `--focus-ring`) already layered over raw neutrals and accents. There is no `[data-theme]` or `prefers-color-scheme` usage anywhere. The persisted `appearance` setting (`'dark' | 'light'`, default dark) is surfaced in Settings with Light disabled — and is never applied to the DOM.

Three facts shape the safety design:

- **The "no magic values" rule is convention, not a gate.** Nothing in the lint chain checks CSS, and roughly six component files already hardcode raw `oklch(...)` values (controls, phototile, titlebar, lightbox, feedback). Today those are style nits; under theming they become functional bugs — spots a theme cannot reach.
- **The renderer is strictly sandboxed** (`sandbox: true`, `contextIsolation: true`) behind a meta-tag CSP (`default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' overlook-thumb: overlook-full:; connect-src 'self' overlook-full:`). A theme system must not widen this.
- **#381 shipped contrast tooling for test lanes** — axe's `color-contrast` rule via axe-playwright over the story and E2E lanes, with the reusable budget verdict in `tests/a11y/budget.ts`. That gates first-party surfaces in CI; it cannot run inside the app at import time.

The epic adds one more requirement with teeth: themes may be authored by an LLM from an exported template, which means import must assume the file is arbitrary — possibly subtly wrong, possibly hostile — and make both outcomes harmless.

## Decision

### 1. Contract shape — JSON token overrides; no CSS mode, full or "sanitized"

A user theme is a **JSON file that maps token names to color values**. There is no full-stylesheet mode and no sanitized-CSS mode, and we record why: CSS sanitization is an open-ended arms race (`url()` and `@import` exfiltration, font-face fetches, `image-set()`, selector bombs that hollow out the UI, future CSS features with network side effects), and nothing in the epic's goal needs it — appearance is colors, and colors fit in JSON. The rejected alternative buys unbounded risk for zero product scope.

Application is injection-proof by construction rather than by filtering: values are **parsed** at validation time into a typed color representation, and the applied stylesheet is **generated by re-serializing those parsed values**. User-supplied bytes never reach a style sheet, an HTML attribute, or `innerHTML` — so there is nothing to sanitize. The CSP does not change for theming; the renderer stays sandboxed; a theme file is data handled like any other untrusted input at the IPC boundary.

### 2. Token layering — dark base, `data-theme` modes, one adopted user layer

Three layers, lowest to highest precedence:

1. **Base (dark)** — the existing `:root` token values, unchanged. Dark remains the definitional theme; every current reference keeps working with zero edits.
2. **First-party mode** — `:root[data-theme='light'] { ... }` overrides in repo CSS (#395 ships the light values). `data-theme` is stamped on `document.documentElement` at renderer boot and on settings change, driven by `appearance`, which gains a third value: `'dark' | 'light' | 'system'` (`'system'` resolves via `nativeTheme.shouldUseDarkColors`, live-updating). First-party themes are reviewed code and may restyle any token, including gradients and shadows. The hardcoded window `backgroundColor` follows the resolved `--surface-window` (#395).
3. **User theme** — a single constructed `CSSStyleSheet` containing one `:root { --token: <re-serialized color>; ... }` rule, appended last via `adoptedStyleSheets`. Apply and revert are one atomic array operation each; no `<style>` element, no inline styles, nothing the CSP or DOM sanitizers need to reason about. A user theme declares `base: 'dark' | 'light'` in its metadata; applying it also applies that base mode underneath, so partial themes inherit sensible values.

Component CSS never special-cases a theme: it keeps consuming `var(--…)` only. That is what makes the whole scheme work — see §7.

### 3. Theme file format, schema, and storage

**Extension `*.overlook-theme.json`**, stored per profile in `userData/themes/` (appearance is app-scope per ADR-0017 §6 — themes follow the profile, not the library). Settings gains `userTheme: string | null` (the applied theme's id); the file itself is the source of truth, settings only points at it.

The zod schema lives in `src/shared/theme/theme-file.ts` and is validated **in the main process** before any renderer contact. Written schema (acceptance 1):

```jsonc
{
  "meta": {
    "name": "Nord-ish",             // 1–80 chars
    "author": "chris",              // 0–120 chars, optional
    "version": "1.0.0",             // semver string
    "base": "dark",                 // 'dark' | 'light' — the mode this theme extends
    "tokensVersion": 1              // integer; the token-schema version it targets
  },
  "tokens": {
    "--accent-iris": "oklch(0.72 0.16 260)",
    "--surface-window": "#101014",
    "--text-body": "oklch(0.93 0.01 260)"
  }
}
```

- **v1 themable surface: color tokens only** — the enumerated `colors.css` set (neutrals, accents and their `-dim` variants, borders, scrim, selection, focus ring, surfaces, text) **excluding** the gradient tokens (`--brand-gradient`, `--brand-gradient-soft`, `--protect-grad`). Spacing, typography, radii, elevation, and motion are not user-themable in v1 — they change layout and readability, and the safety story ("every value is a single color") is worth more than the flexibility. The exact list is one exported array in the schema module; the template (§6) is generated from it, so contract and documentation cannot drift.
- **Value rule:** every value must parse as a single CSS `<color>` (hex, `rgb()`, `hsl()`, `oklch()`, `color()`). Parsing happens in the schema module with a pure parser — no `var()`, no `calc()`, no functions with URLs, no strings passed through unparsed.
- **Unknown tokens warn-and-skip** (reported by name, not applied): a theme written against an older token schema keeps working as the surface grows; the file round-trips unmodified. A theme where *nothing* matches fails validation outright.
- **Error shape:** validation returns the parsed theme or a `{ path, message }[]` list — rendered verbatim in the import dialog (#396), so an LLM-authored file that got one token wrong produces a fixable message, not a mystery failure.

### 4. Contrast guardrails — warn at AA, block only the unusable

Import-time validation computes WCAG 2.x contrast ratios with a **pure relative-luminance module** in `src/shared/theme/` (unit-tested against published WCAG examples — axe cannot run at import time; this is the same math without the browser). Checked pairs are enumerated next to the token list: each `--text-*` against each `--surface-*`, `--text-on-accent` against each accent, `--focus-ring` against `--surface-window`.

- **Warn below 4.5:1** on text/surface pairs (AA for normal text): shown in the import/preview UI with the failing pairs and ratios; the user may apply anyway — a personal theme is their aesthetic choice, and the revert path (§5) does not depend on their restraint.
- **Hard-block below 1.5:1** on any `--text-body`/surface pair: at that point text is functionally invisible and the theme is indistinguishable from hostile; the file is rejected with the pair named.
- **First-party themes meet a higher bar:** both dark and light must pass the existing axe `color-contrast` budget lanes (stories run under both themes per the epic's exit criteria) — CI-gated via the #381 machinery, not the import checker.

### 5. Recovery — revert must not need the renderer's cooperation

Ordered from least to most drastic; the first two are automatic:

1. **Preview-first apply:** the theme manager applies to the live window without persisting; persisting `userTheme` happens only on explicit confirm.
2. **Health-gated persist:** after apply, the renderer must acknowledge over IPC within a short deadline; a missing ack, a `render-process-gone`, or an `unresponsive` event auto-reverts the adopted sheet and does not persist. A persisted theme that later fails at boot (file deleted, validation fails against a newer schema) is skipped with a notice — never a crash loop.
3. **Native escape hatches, zero renderer involvement:** an app-menu item (**View → Reset Appearance**) and a `--reset-theme` CLI flag both clear `userTheme` and reset `appearance` to dark from the main process. Because user themes are colors-only (§3), the worst reachable state is unreadable text — and both hatches live in native chrome, outside the themed surface.

### 6. Template export and LLM theming (#397)

The theme manager exports an **annotated template**: a valid `*.overlook-theme.json` containing every themable token with its current effective value, plus a parallel `docs` map (token → one-line description of what it colors) and the §4 pair list — generated from the schema module, so the exported surface is exactly the accepted surface. **Round-trip guarantee:** exporting and re-importing without edits validates cleanly and applies as a no-op; this is a unit test in #397. A wiki page documents the LLM workflow (paste template + instructions, import the result) and states the safety posture plainly: whatever the LLM returns, import validates it like any untrusted file — §3 and §4 are the contract, not trust in the generator. An example theme ships on the wiki alongside it.

### 7. The guardrail becomes a gate

Theming works only if component CSS references tokens — every hardcoded color is a spot no theme can reach. Accordingly:

- **The "no magic values" convention becomes an automated check** (with #395): a lint step that rejects raw color literals (`#hex`, `rgb`/`hsl`/`oklch(...)`) in renderer CSS outside `styles/tokens/`, wired into the existing lint chain per ADR-0001. Local component custom properties must be defined in terms of tokens (`color-mix()` over `var(--…)` is fine; raw literals are not).
- **The known violations are cleaned up in the same slice** (controls, phototile, titlebar, lightbox, feedback CSS) — they are pre-existing bugs that theming exposes.
- **Source-of-truth relationship:** `src/renderer/src/styles/tokens/` is the runtime and schema source of truth; `design/handoff/tokens/` remains the design package. The copies have already drifted cosmetically (formatting, a font path) — a parity check is recorded as a deferral, not blocking this epic.

## Consequences

- **Easier:** the semantic alias layer already in `colors.css` is exactly the theming surface a user wants; `adoptedStyleSheets` gives atomic apply/revert; validation-in-main matches every other untrusted-input boundary in the app; the axe budget machinery gates first-party themes with no new infrastructure.
- **Harder:** the lint gate will surface every hardcoded color at once (one-time cleanup cost in #395); `appearance: 'system'` adds a live `nativeTheme` listener to the settings flow; the light theme must clear the same contrast budgets as dark before it can ship enabled.
- **Deliberately excluded, with rationale recorded:** any CSS import mode (unbounded sanitization risk, no product need); user-themable spacing/typography/motion (layout and readability risk — revisit only with a concrete need and a per-token safety argument); marketplace/sharing (epic out-of-scope).
- **Deferred, with owners:** handoff↔renderer token parity check → backlog issue out of this epic; gradient tokens joining the themable surface → revisit after v1 ships (needs a gradient grammar, which reopens the parsing question); in-app LLM generation → the #379 assistant epic, against this contract unchanged.
- **Revisit when:** the token schema needs a breaking rename (bump `tokensVersion`, add a migration map in the schema module — the warn-and-skip rule covers additive growth only).
